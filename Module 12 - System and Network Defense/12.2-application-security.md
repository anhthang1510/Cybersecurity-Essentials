# 12.2 - Application Security

## Tổng quan

Bảo mật ứng dụng bảo vệ phần mềm trong suốt vòng đời từ phát triển đến ngừng hoạt động. Tấn công ứng dụng thường nhắm vào dữ liệu trong cơ sở dữ liệu, khai thác input không được kiểm tra, hoặc cài mã độc vào quá trình cập nhật phần mềm. Bài này đề cập quy trình phát triển an toàn, các kỹ thuật coding bảo mật, kiểm tra tính toàn vẹn dữ liệu, và quản lý mối đe dọa ứng dụng.

## Ý chính

**Vòng đời phát triển ứng dụng (Software Development Lifecycle):**

| Môi trường | Mục đích | Mức bảo mật |
|-----------|---------|------------|
| Development | Viết code, kiểm thử, debug | Thấp nhất — linh hoạt để phát triển |
| Staging | Kiểm tra trong điều kiện gần giống production | Trung bình — mô phỏng môi trường thực |
| Production | Hệ thống đang chạy thực tế | Cao nhất — áp dụng đầy đủ security settings |

Provisioning là tạo mới hoặc cập nhật phần mềm. Deprovisioning là gỡ bỏ phần mềm khi không còn cần thiết. Version control giúp theo dõi thay đổi và ngăn ghi đè code trong môi trường nhóm.

**Kỹ thuật bảo mật khi coding:**

| Kỹ thuật | Mục đích |
|---------|---------|
| Normalization | Chuyển đổi chuỗi đầu vào về dạng đơn giản nhất để phát hiện input độc hại |
| Stored procedures | Nhóm lệnh SQL được biên dịch sẵn; giảm tấn công SQL injection |
| Obfuscation | Che giấu dữ liệu thực bằng ký tự ngẫu nhiên, khó đảo ngược |
| Camouflage | Thay thế dữ liệu nhạy cảm bằng dữ liệu giả thực tế |
| Code reuse | Tái sử dụng code tiết kiệm thời gian nhưng cần cẩn thận với lỗ hổng kế thừa |
| Third-party libraries/SDKs | Lỗ hổng trong thư viện bên thứ ba ảnh hưởng tất cả ứng dụng sử dụng nó |

**Kiểm tra và bảo đảm tính toàn vẹn:**

- **Input validation**: Kiểm soát dữ liệu đầu vào để ngăn tấn công injection và tràn bộ đệm.
- **Validation rules**: Kiểm tra kích thước, định dạng, tính nhất quán, phạm vi và check digit.
- **Integrity check**: Dùng hash function (MD5, SHA-256, SHA-512) để tạo "ảnh chụp" dữ liệu và so sánh sau đó phát hiện thay đổi.
- **Checksum**: Ví dụ cụ thể của hash function kiểm tra tính toàn vẹn khi truyền file.
- **Version control**: Ngăn hai người chỉnh sửa cùng một file cùng lúc.

**Các thực hành bảo mật ứng dụng khác:**

- **Code signing**: Ký điện tử trên file thực thi để xác nhận danh tính tác giả và phần mềm chưa bị thay đổi.
- **Secure cookies**: Dùng HTTPS khi truyền cookie để ngăn đánh cắp qua kênh không mã hóa.

| Thuật ngữ | Định nghĩa |

## Ghi chú chi tiết

**Tại sao staging environment quan trọng?**

Nếu triển khai thẳng từ development lên production, code chưa được kiểm tra trong điều kiện thực tế và có thể phát sinh lỗi nghiêm trọng. Staging environment phản ánh sát nhất môi trường production, giúp kiểm tra security settings, performance, và tương thích trước khi chạy thực tế. Sửa lỗi ở staging nhanh hơn và ít tốn kém hơn nhiều so với sửa lỗi trên hệ thống đang chạy.

**Input validation và tấn công tự động:**

Ví dụ về tấn công tự động: một trang đăng ký newsletter gửi email xác nhận với link URL. Kẻ tấn công sửa đổi URL đó để thay đổi thông tin người dùng khi họ nhấp vào. Bằng cách tự động hóa, hacker có thể gửi hàng nghìn đăng ký giả trong thời gian ngắn. Input validation kiểm tra từng trường dữ liệu theo quy tắc định trước để ngăn dữ liệu giả mạo đi vào hệ thống.

**Hash function trong thực tế:**

Khi tải file từ internet (ví dụ firmware của router), nhà sản xuất thường cung cấp hash SHA-256 của file. Sau khi tải về, chạy lại hash trên file đã tải và so sánh với giá trị gốc. Nếu khớp, file không bị thay đổi trong quá trình tải. Nếu không khớp, file có thể đã bị chỉnh sửa (man-in-the-middle) hoặc bị lỗi trong quá trình truyền.

**Third-party libraries — rủi ro lan rộng:**

Khi một lỗ hổng được phát hiện trong thư viện nguồn mở phổ biến (ví dụ Log4j năm 2021), mọi ứng dụng sử dụng thư viện đó đều bị ảnh hưởng đồng thời, kể cả các hệ thống quan trọng của chính phủ và doanh nghiệp lớn. Kiểm soát third-party dependencies là một phần quan trọng của bảo mật ứng dụng hiện đại.

**Countermeasures trong application domain:**

| Mối đe dọa | Biện pháp đối phó |
|-----------|-----------------|
| Bảo mật cơ sở vật chất | Chính sách kiểm soát truy cập cho nhân viên và khách |
| Mất tính sẵn sàng | Kế hoạch business continuity và disaster recovery |
| Lỗ hổng phần mềm | Chính sách vá và cập nhật định kỳ |
| Truy cập trái phép | Multi-factor authentication; theo dõi log |
| Rò rỉ dữ liệu | Phân loại và sao lưu dữ liệu |
| Lỗi phát sinh | Kiểm thử kỹ trước khi triển khai |

## Ví dụ thực tế

**Ví dụ Code Signing:**

Khi tải phần mềm từ trang chính thức, Windows hiển thị thông báo "Publisher: [Tên công ty]" với chữ ký số xác nhận. Nếu thấy "Unknown Publisher", phần mềm có thể chưa được ký hoặc chữ ký không hợp lệ, đây là dấu hiệu cảnh báo. Doanh nghiệp có thể dùng code signing để kiểm soát phần mềm nội bộ.

**Ví dụ Secure Cookie:**

Trang e-banking ghi nhớ phiên đăng nhập qua cookie. Nếu cookie được gửi qua HTTP (không mã hóa), kẻ tấn công trên cùng mạng wifi có thể chặn và đánh cắp cookie đó, sau đó giả mạo phiên của người dùng (session hijacking). Dùng HTTPS và thuộc tính `Secure` trên cookie ngăn chặn điều này.

## Cần ghi nhớ

- **Normalization, stored procedures và input validation** là ba lớp bảo vệ chính chống SQL injection và tấn công dữ liệu.
- **Hash function tạo "dấu vân tay" dữ liệu** — thay đổi dù một bit cũng cho kết quả hash hoàn toàn khác.
- **Code signing xác nhận nguồn gốc và tính toàn vẹn** của phần mềm trước khi cài đặt.
- **Third-party libraries là điểm yếu tiềm ẩn** — một lỗ hổng ảnh hưởng hàng trăm ứng dụng.
- **Staging environment là bắt buộc** trước khi triển khai lên production.
- **OWASP Top 10** là tài liệu tham khảo tiêu chuẩn về các lỗ hổng ứng dụng web phổ biến nhất.
