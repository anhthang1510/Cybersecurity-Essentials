# 13.3 - Account Management

## Tổng quan

Quản lý tài khoản là cách tổ chức kiểm soát ai có quyền truy cập gì vào hệ thống. Bài này đề cập các loại tài khoản, cách quản lý tài khoản đặc quyền, quyền file trong hệ thống, chính sách tài khoản Windows, cơ chế HMAC, và các giao thức xác thực phổ biến.

## Ý chính

**Các loại tài khoản:**

| Loại tài khoản | Mục đích | Rủi ro |
|--------------|---------|-------|
| User account | Truy cập thông thường hàng ngày | Thấp nếu quản lý đúng |
| Service account | Tương tác với hệ điều hành, chạy dịch vụ | Trung bình — thường có quyền hệ thống |
| Guest account | Người dùng tạm thời | Cao nếu không giới hạn đúng |
| Administrator account | Quản trị hệ thống | Cao nhất — mục tiêu chính của tấn công |

Tài khoản mặc định (như "administrator" hoặc "guest") là rủi ro bảo mật vì kẻ tấn công biết tên và cài đặt mặc định. Thay thế hoặc vô hiệu hóa tài khoản mặc định là bước bảo mật cơ bản.

**Quản lý tài khoản đặc quyền (Privileged Accounts):**

Tài khoản đặc quyền là mục tiêu số một vì chúng có quyền truy cập không giới hạn vào hệ thống. Thực hành tốt nhất:

- Xác định và giảm số lượng tài khoản đặc quyền.
- Áp dụng least privilege — chỉ quyền cần thiết cho nhiệm vụ.
- Thu hồi quyền ngay khi nhân viên rời công ty hoặc đổi vị trí.
- Không dùng tài khoản chia sẻ cho nhiều admin.
- Lưu trữ mật khẩu an toàn.
- Tự động đổi mật khẩu tài khoản đặc quyền mỗi 30-60 ngày.
- Ghi lại tất cả phiên đặc quyền (privileged session recording).
- Đổi mật khẩu nhúng trong script và service account.
- Log mọi hoạt động của người dùng.
- Cảnh báo khi có hành vi bất thường.
- Vô hiệu hóa tài khoản đặc quyền không hoạt động.
- Dùng MFA cho tất cả truy cập quản trị.

**Quyền File (File Access Control):**

Quyền giới hạn truy cập file và folder theo người dùng hoặc nhóm.

| Mức quyền | Được làm gì |
|----------|-----------|
| Full Control | Xem nội dung, thay đổi, xóa file hiện có, tạo mới, chạy chương trình |
| Modify | Thay đổi và xóa file hiện có, không tạo mới |
| Read and Execute | Xem nội dung file hiện có và chạy chương trình |
| Write | Tạo file mới và thay đổi file hiện có |
| Read | Xem nội dung file và folder, mở file và folder |

Deny permission ghi đè mọi quyền khác — kể cả Admin không thể truy cập nếu bị Deny rõ ràng.

**Quy tắc kế thừa quyền:**

| Tình huống | Quyền kế thừa |
|-----------|--------------|
| Di chuyển trong cùng volume | Giữ quyền gốc |
| Sao chép trong cùng volume | Kế thừa quyền mới của folder đích |
| Di chuyển sang volume khác | Kế thừa quyền mới |
| Sao chép sang volume khác | Kế thừa quyền mới |

**Chính sách tài khoản Windows:**

Trong môi trường Active Directory, admin cấu hình chính sách miền áp dụng cho tất cả thành viên. Trên máy độc lập, dùng Local Security Policy (secpol.msc).

Chính sách mật khẩu ví dụ:
- Thay mỗi 90 ngày.
- Dùng ít nhất 1 ngày trước khi đổi lại.
- Tối thiểu 8 ký tự, 3/4 loại ký tự.
- Không tái sử dụng trong 24 mật khẩu gần nhất.

Chính sách khóa tài khoản ví dụ:
- Khóa sau 5 lần đăng nhập sai.
- Khóa 30 phút.
- Reset bộ đếm sau 30 phút.

Audit policy ghi log các sự kiện: đăng nhập, quản lý tài khoản, truy cập đối tượng, thay đổi chính sách, sử dụng đặc quyền.

**Authentication Management:**

- **SSO (Single Sign-On)**: Một bộ credential truy cập nhiều ứng dụng.
- **OAuth**: Tiêu chuẩn cho phép thông tin tài khoản dùng bởi service bên thứ ba (như "Đăng nhập bằng Google").
- **Password vault**: Lưu trữ tất cả mật khẩu mã hóa; chỉ cần nhớ một mật khẩu master.
- **KBA (Knowledge-Based Authentication)**: Câu hỏi bảo mật để đặt lại mật khẩu.

**HMAC (Hash-Based Message Authentication Code):**

HMAC dùng khóa mã hóa kết hợp hàm hash để xác thực người dùng web. Web service thông thường dùng basic authentication — không mã hóa username/password. Với HMAC:
1. Người dùng gửi private key identifier và HMAC.
2. Server tra cứu private key của người dùng.
3. Server tạo HMAC của mình.
4. So sánh HMAC của người dùng với HMAC của server — nếu khớp, xác thực thành công.

Ứng dụng HMAC: IPsec dùng HMAC để xác thực nguồn và kiểm tra tính toàn vẹn gói tin; Cisco IOS dùng HMAC với secret key cho routing protocol; MD5 và SHA-1 trong chế độ HMAC trong IPsec gateway.

**Giao thức xác thực:**

| Giao thức | Cơ chế | Bảo mật |
|----------|-------|--------|
| PAP | Username và password plaintext | Thấp — truyền rõ |
| CHAP | Hash one-way; server gửi challenge; client tính hash | Cao hơn PAP |
| RADIUS | Kết hợp authentication và authorization; chỉ mã hóa mật khẩu | Trung bình |
| TACACS+ | Tách authentication, authorization, accounting; mã hóa toàn bộ | Cao |
| Kerberos | Ticket-based; mã hóa mạnh; xác thực hai chiều | Rất cao |

**Hash functions trong bảo mật:**

- Dùng SHA-256 hoặc cao hơn — SHA-1 và MD5 đã có lỗ hổng đã biết.
- IPsec, routing protocol auth, certificate PKI đều dùng hash function.
- Hash chỉ phát hiện thay đổi ngẫu nhiên, không chống được thay đổi có chủ đích (cần thêm secret key = HMAC).

## Thuật ngữ quan trọng

| Thuật ngữ | Định nghĩa |
|-----------|-----------|
| Privileged account | Tài khoản có quyền nâng cao không giới hạn để quản trị hệ thống |
| Least privilege | Chỉ cấp quyền tối thiểu cần thiết để thực hiện nhiệm vụ |
| File permission | Quy tắc kiểm soát quyền truy cập file và folder |
| Deny permission | Quyền từ chối ghi đè mọi quyền khác, kể cả Admin |
| HMAC | Hash-Based Message Authentication Code — xác thực dùng hash + secret key |
| SSO | Single Sign-On — một lần đăng nhập truy cập nhiều hệ thống |
| PAP | Password Authentication Protocol — mật khẩu plaintext |
| CHAP | Challenge Handshake Authentication Protocol — hash one-way |
| RADIUS | Remote Authentication Dial-In User Service — xác thực tập trung |
| TACACS+ | Terminal Access Controller Access Control System Plus — xác thực Cisco |
| Kerberos | Giao thức xác thực mạng dùng ticket và mã hóa mạnh |
| KBA | Knowledge-Based Authentication — câu hỏi bảo mật |
| Password vault | Công cụ lưu trữ mật khẩu mã hóa; truy cập bằng một master password |
| Audit policy | Chính sách định nghĩa sự kiện nào được ghi log trên hệ thống |

## Ghi chú chi tiết

**Tại sao tài khoản service nguy hiểm?**

Service account chạy dịch vụ hệ thống nên thường cần quyền hệ thống cao. Nhiều tổ chức dùng mật khẩu tĩnh không bao giờ đổi cho service account. Nếu kẻ tấn công lấy được mật khẩu đó (qua memory dump hoặc config file), họ có persistent access với quyền rất cao mà không ai phát hiện vì không có người dùng tương tác.

**CHAP hoạt động chi tiết:**

1. Client yêu cầu kết nối.
2. Server gửi challenge (chuỗi ngẫu nhiên).
3. Client kết hợp challenge với mật khẩu, tính hash, gửi lại.
4. Server thực hiện phép tính tương tự và so sánh.
5. Nếu khớp, xác thực thành công.

CHAP không bao giờ gửi mật khẩu thực qua mạng. Tuy nhiên, CHAP không chống được Man-in-the-Middle attacks phức tạp. CHAP cũng xác thực lại định kỳ trong suốt phiên.

**RADIUS vs TACACS+ — Khi nào dùng cái nào:**

RADIUS phổ biến hơn và là chuẩn mở (RFC). Dùng RADIUS khi: tổ chức không phải toàn Cisco; cần xác thực cho VPN hoặc Wi-Fi với nhiều loại thiết bị khác nhau.

TACACS+ độc quyền Cisco hơn nhưng tốt hơn cho quản lý thiết bị mạng: tách authentication/authorization/accounting hoàn toàn (linh hoạt hơn); mã hóa toàn bộ gói tin (bảo mật hơn); có thể định nghĩa quyền theo từng lệnh IOS (chi tiết hơn).

**Kerberos trong Windows domain:**

Kerberos là giao thức xác thực mặc định trong Windows Active Directory. Khi đăng nhập vào domain, bạn nhận Ticket Granting Ticket (TGT) từ KDC (Key Distribution Center). Khi cần truy cập tài nguyên (ví dụ file server), TGT được dùng để xin service ticket — không cần nhập lại mật khẩu. Đây chính là cơ chế SSO trong Windows domain.

## Ví dụ thực tế

**Ví dụ quyền file cho dự án nghiên cứu:**

| Đối tượng | Quyền | Lý do |
|---------|------|------|
| Research Team | Read | Đọc tài liệu tham khảo |
| Research Leads | Read and Write | Tạo và cập nhật báo cáo |
| IT Admin | Full Control | Quản lý quyền và cấu hình |
| Auditor | Read | Xem xét nhưng không thay đổi |
| Nhân viên khác | Deny | Dữ liệu nhạy cảm, không được phép xem |

**Ví dụ quy trình onboarding/offboarding:**

Onboarding: tạo tài khoản với quyền tối thiểu theo vai trò → cấp badge truy cập theo vị trí → đăng ký thiết bị → đào tạo bảo mật.

Offboarding: vô hiệu hóa tài khoản ngay → thu hồi badge → xóa certificate → remote wipe thiết bị công ty → chuyển dữ liệu công việc → audit account.

## Cần ghi nhớ

- **Deny ghi đè mọi quyền khác** — kể cả tài khoản Admin không thể truy cập nếu bị Deny.
- **Tài khoản mặc định cần được thay thế hoặc vô hiệu hóa** — kẻ tấn công biết mặc định.
- **HMAC = hash + secret key** — xác thực tính toàn vẹn VÀ nguồn gốc cùng lúc.
- **TACACS+ mã hóa toàn bộ; RADIUS chỉ mã hóa mật khẩu** — đây là khác biệt bảo mật quan trọng.
- **SHA-256 trở lên** — SHA-1 và MD5 đã có lỗ hổng.
- **Service account mật khẩu tĩnh là rủi ro lớn** — nên tự động xoay vòng định kỳ.
