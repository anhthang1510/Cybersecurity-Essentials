# 18.3 - Integrity and Authenticity

## Tổng quan

Integrity đảm bảo dữ liệu không bị thay đổi ngoài ý muốn, còn authenticity giúp xác nhận dữ liệu hoặc thông điệp đến từ nguồn hợp lệ. Cryptography hỗ trợ hai mục tiêu này bằng hash, MAC/HMAC và digital signatures.

## Ý chính

- Integrity trả lời câu hỏi “dữ liệu có bị thay đổi không?”.
- Authenticity trả lời câu hỏi “nguồn gửi có đúng là bên được tin tưởng không?”.
- Hash giúp phát hiện thay đổi nhưng không tự xác thực nguồn gửi.
- HMAC dùng secret key để xác thực thông điệp và kiểm tra toàn vẹn.
- Digital signature dùng private key để ký và public key để kiểm tra.
- Non-repudiation liên quan khả năng chứng minh bên ký khó phủ nhận hành động ký.

## Thuật ngữ quan trọng

| Thuật ngữ | Ý nghĩa |
|---|---|
| Integrity | Tính toàn vẹn, đảm bảo dữ liệu không bị sửa trái phép. |
| Authenticity | Tính xác thực, đảm bảo nguồn gửi hoặc danh tính là hợp lệ. |
| Message digest | Giá trị hash đại diện cho dữ liệu đầu vào. |
| MAC | Message Authentication Code, xác thực thông điệp bằng secret key. |
| HMAC | MAC dựa trên hash function và secret key. |
| Digital signature | Chữ ký số dùng private key để ký dữ liệu. |
| Non-repudiation | Khả năng giảm nguy cơ bên ký phủ nhận hành động đã thực hiện. |

## Ghi chú chi tiết

**Integrity không tự đồng nghĩa với authenticity:**

Nếu bạn tải file và so sánh hash từ cùng một nguồn bị attacker kiểm soát, attacker có thể thay cả file lẫn hash. Integrity chỉ đáng tin hơn khi digest hoặc chữ ký đến từ nguồn đáng tin cậy.

**HMAC:**

HMAC kết hợp hash function với secret key. Nếu bên nhận tính lại HMAC và kết quả khớp, họ biết dữ liệu không bị sửa và người tạo HMAC có secret key hợp lệ. HMAC phù hợp cho API request signing hoặc xác thực message giữa hệ thống cùng chia sẻ secret.

**Digital signature:**

Digital signature dùng private key để ký digest của dữ liệu. Bất kỳ ai có public key tương ứng đều có thể kiểm tra chữ ký. Nếu public key được xác thực qua certificate/PKI, chữ ký số hỗ trợ xác minh nguồn gốc tốt hơn.

**Non-repudiation:**

Non-repudiation không chỉ là kỹ thuật cryptography; nó còn phụ thuộc vào bảo vệ private key, logging, policy và quy trình pháp lý. Nếu private key bị chia sẻ hoặc bị lộ, giá trị chứng minh sẽ yếu đi.

## Ví dụ thực tế

**Ví dụ 1: Kiểm tra file tải về**

Nhà phát hành công bố SHA-256 hash cho file ISO. Sau khi tải, người dùng tính hash local và so sánh. Nếu khác nhau, file có thể bị lỗi hoặc bị thay đổi.

**Ví dụ 2: Ký phần mềm**

Nhà phát triển ký phần mềm bằng private key. Hệ điều hành kiểm tra chữ ký bằng certificate của nhà phát hành. Nếu chữ ký hợp lệ, người dùng có thêm cơ sở tin rằng phần mềm chưa bị sửa sau khi ký.

## Cần ghi nhớ

- **Integrity kiểm tra thay đổi**: hash là công cụ phổ biến.
- **Authenticity cần xác thực nguồn**: HMAC hoặc digital signature hỗ trợ mục tiêu này.
- **Hash đơn thuần không chứng minh người gửi**: ai cũng có thể tính hash.
- **Digital signature cần private key an toàn**: lộ private key làm mất trust.
- **Non-repudiation cần cả kỹ thuật và quy trình**: không chỉ dựa vào thuật toán.
