# 18.4 - Using Hashes

## Tổng quan

Hash function nhận dữ liệu đầu vào có độ dài bất kỳ và tạo digest có độ dài cố định. Hash được dùng rộng rãi để kiểm tra integrity, lưu password an toàn hơn, tạo fingerprint cho file và hỗ trợ digital signature.

## Ý chính

- Hash là hàm một chiều, không thiết kế để giải mã về dữ liệu gốc.
- Thay đổi nhỏ trong input nên tạo digest khác biệt rõ rệt.
- Hash tốt cần khó tìm collision trong thực tế.
- Password không nên lưu plaintext; nên dùng password hashing có salt và thuật toán phù hợp.
- Hash file giúp phát hiện thay đổi hoặc lỗi tải xuống.
- Digital signature thường ký hash của dữ liệu thay vì ký toàn bộ dữ liệu lớn.

## Thuật ngữ quan trọng

| Thuật ngữ | Ý nghĩa |
|---|---|
| Hash function | Hàm tạo digest cố định từ input. |
| Digest | Giá trị đầu ra của hash function. |
| One-way function | Hàm dễ tính theo chiều xuôi nhưng khó đảo ngược. |
| Collision | Hai input khác nhau tạo cùng digest. |
| Salt | Giá trị ngẫu nhiên thêm vào password trước khi hash để chống rainbow table. |
| Rainbow table | Bảng tra cứu hash đã tính sẵn cho nhiều password phổ biến. |
| Password hashing | Cơ chế hash password chuyên dụng, thường chậm và có salt. |
| Checksum | Giá trị kiểm tra lỗi, không luôn đủ mạnh cho bảo mật. |

## Ghi chú chi tiết

**Hash không phải encryption:**

Encryption có thể giải mã nếu có key. Hash thì không có quá trình “giải hash” hợp lệ. Nếu muốn xác minh password, hệ thống hash password người dùng nhập và so sánh với hash đã lưu.

**Collision resistance:**

Một hash function bảo mật cần khó tìm hai input khác nhau có cùng digest. Các thuật toán cũ như MD5 và SHA-1 không còn phù hợp cho mục tiêu bảo mật hiện đại vì đã có tấn công collision thực tế.

**Salt trong password hashing:**

Nếu hai người dùng có cùng password và không dùng salt, hash của họ giống nhau. Salt làm mỗi hash khác nhau, khiến rainbow table kém hiệu quả hơn. Password hashing hiện đại cũng nên chậm có chủ đích để giảm tốc độ brute force.

**Hash trong digital signature:**

Thay vì ký toàn bộ file lớn, hệ thống tính hash của file rồi ký digest. Khi kiểm tra, bên nhận tính lại hash và xác minh chữ ký trên digest.

## Ví dụ thực tế

**Ví dụ 1: Xác minh bản tải xuống**

Bạn tải file cài đặt và so sánh SHA-256 digest với giá trị nhà phát hành công bố trên kênh đáng tin cậy. Nếu digest khớp, file ít có khả năng bị thay đổi trong quá trình tải.

**Ví dụ 2: Lưu password**

Website không nên lưu password plaintext. Thay vào đó, hệ thống lưu password hash có salt bằng thuật toán chuyên dụng. Khi user đăng nhập, hệ thống hash password nhập vào cùng salt và so sánh kết quả.

## Cần ghi nhớ

- **Hash là một chiều**: không giải mã hash để lấy dữ liệu gốc.
- **MD5/SHA-1 không phù hợp cho bảo mật mới**: nên dùng thuật toán hiện đại hơn.
- **Salt chống rainbow table**: cùng password nhưng salt khác sẽ tạo hash khác.
- **Password hashing cần chậm có chủ đích**: để giảm hiệu quả brute force.
- **Hash cần nguồn đáng tin cậy**: hash bị công bố trên nguồn bị compromise không còn đáng tin.
