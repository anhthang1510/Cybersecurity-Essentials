# 4.2 - Enterprise Services

## Tổng quan

Phần này trình bày cách các dịch vụ doanh nghiệp như web browsing, HTTP/HTTPS, email, web applications và databases có thể bị khai thác. Đây là những dịch vụ gần như không thể tắt trong môi trường doanh nghiệp, vì vậy mục tiêu thực tế là cấu hình an toàn, giám sát tốt và giảm thiểu tác động khi có tấn công.

## Ý chính

- Web browser là điểm tiếp xúc lớn giữa người dùng và internet, nên thường bị dùng để phát tán malware.
- Web attack có thể bắt đầu từ một website bị compromise, sau đó redirect người dùng tới exploit kit hoặc malware server.
- HTTP status codes và server logs giúp security analyst nhận diện scan, lỗi ứng dụng hoặc redirect bất thường.
- Malicious iFrames, HTTP 302 cushioning và domain shadowing là các kỹ thuật phổ biến trong web-based attacks.
- Email là kênh tấn công lớn với attachment-based attacks, spoofing, spam, open relay và homoglyph tricks.
- Web-exposed databases dễ bị code injection và SQL injection nếu input validation yếu.
- XSS là tấn công client-side, chạy script độc hại trong browser của nạn nhân.

## Ghi chú chi tiết

### HTTP/HTTPS và web-based attacks

Doanh nghiệp cần web access để làm việc, vì vậy không thể đơn giản chặn toàn bộ web browsing. Security analyst cần hiểu cách một cuộc tấn công web thường diễn ra:

1. Người dùng truy cập một website đã bị compromise hoặc một link độc hại.
2. Trang đó redirect người dùng qua một hoặc nhiều server trung gian.
3. Browser đến trang chứa exploit kit hoặc mã độc.
4. Exploit kit kiểm tra browser, OS, Java, Flash hoặc plugin cũ để tìm lỗ hổng.
5. Nếu tìm được lỗ hổng, mã khai thác được tải xuống và chạy.
6. Máy nạn nhân tải payload cuối cùng, ví dụ malware hoặc downloader.

HTTPS giúp bảo vệ đường truyền nhưng không tự động làm website an toàn. Một website độc hại vẫn có thể dùng HTTPS hợp lệ. Vì vậy cần kết hợp TLS, browser hardening, patching, DNS/web filtering, IDS/IPS và user awareness.

### HTTP status codes và log analysis

Server connection logs có thể giúp phân tích scan hoặc attack. Các nhóm mã thường gặp:

- 1xx: thông tin tạm thời.
- 2xx: request thành công.
- 3xx: redirect, cần chú ý nếu có chuỗi redirect bất thường.
- 4xx: lỗi phía client, có thể xuất hiện nhiều khi có scan hoặc thử đường dẫn không tồn tại.
- 5xx: lỗi phía server, có thể liên quan đến lỗi ứng dụng hoặc khai thác gây crash.

Khi điều tra, analyst nên nhìn vào URL, user-agent, referrer, source IP, tần suất lỗi, pattern truy vấn và chuỗi redirect.

### Common HTTP exploits

Malicious iFrame dùng thẻ iFrame để nhúng nội dung từ server độc hại vào trang hợp lệ. Vì iFrame có thể rất nhỏ hoặc khó thấy, người dùng không nhận ra trang đang tải thêm nội dung nguy hiểm.

HTTP 302 cushioning lạm dụng redirect hợp lệ. Người dùng có thể đi qua nhiều redirect trước khi tới exploit landing page, làm quá trình truy vết khó hơn.

Domain shadowing xảy ra khi attacker chiếm tài khoản quản lý domain và tạo nhiều subdomain độc hại dưới domain hợp lệ. Vì parent domain có thể có uy tín, subdomain độc hại dễ vượt qua một số kiểm tra đơn giản.

Countermeasures:

- Cập nhật OS, browser và plugin.
- Dùng web proxy hoặc secure web gateway để chặn site độc hại.
- Áp dụng OWASP best practices khi phát triển web applications.
- Hạn chế dùng iFrame không cần thiết và kiểm soát third-party content.
- Bảo vệ tài khoản quản lý domain bằng strong password và MFA.
- Đào tạo người dùng nhận biết redirect, link rút gọn và domain lạ.

### Email threats

Email là backbone của giao tiếp doanh nghiệp nên là kênh tấn công rất phổ biến. Các rủi ro chính:

- Attachment-based attacks: file đính kèm chứa macro, script hoặc malware.
- Email spoofing: giả mạo sender để lừa chuyển tiền, nhập mật khẩu hoặc mở link.
- Spam: gửi hàng loạt email quảng cáo hoặc email độc hại.
- Open mail relay: SMTP server cấu hình sai cho phép attacker gửi spam/malware qua server của tổ chức.
- Homoglyphs: dùng ký tự giống nhau để tạo domain hoặc URL đánh lừa mắt người dùng.

Biện pháp giảm thiểu:

- Cập nhật mail server và email security gateway.
- Bật SPF, DKIM, DMARC cho domain email.
- Chặn open relay.
- Sandboxing file đính kèm đáng ngờ.
- URL filtering và cảnh báo domain giả mạo.
- MFA cho tài khoản email, đặc biệt là tài khoản quản trị và tài khoản lãnh đạo.
- Huấn luyện người dùng kiểm tra sender, domain, attachment và yêu cầu bất thường.

### Web-exposed databases và injection

Web applications thường kết nối tới relational database. Nếu input từ người dùng không được kiểm tra, attacker có thể chèn lệnh hoặc SQL query độc hại.

Code injection xảy ra khi attacker khiến ứng dụng thực thi lệnh hoặc code không mong muốn, thường do input validation yếu hoặc xử lý dữ liệu nguy hiểm.

SQL injection xảy ra khi input được ghép trực tiếp vào SQL query. Hậu quả có thể gồm:

- Đọc dữ liệu nhạy cảm.
- Sửa hoặc xóa dữ liệu.
- Bỏ qua đăng nhập.
- Liệt kê bảng, cột, user hoặc version database.
- Trong một số trường hợp, thực thi lệnh trên hệ điều hành.

Countermeasures:

- Prepared statements hoặc parameterized queries.
- Input validation theo allowlist.
- Output encoding khi hiển thị dữ liệu.
- Không dùng tài khoản database có quyền quá rộng.
- Log truy vấn bất thường và lỗi database.
- Kiểm thử bảo mật web app theo OWASP Top 10.

### Client-side scripting và XSS

XSS xảy ra khi attacker chèn script để browser của nạn nhân chạy trong ngữ cảnh của website tin cậy. Script có thể đánh cắp session token, thay đổi nội dung trang, gửi request thay người dùng hoặc chuyển hướng sang site độc hại.

Hai loại chính:

- Stored XSS: script được lưu trên server, ảnh hưởng nhiều người truy cập.
- Reflected XSS: script nằm trong link hoặc request, nạn nhân phải click hoặc mở URL độc hại.

Countermeasures:

- Validate input và encode output.
- Dùng Content Security Policy (CSP).
- Dùng cookie Secure, HttpOnly và SameSite khi phù hợp.
- Dùng framework có cơ chế escape mặc định.
- Dùng IPS/WAF để phát hiện payload phổ biến.
- Đào tạo người dùng tránh link đáng ngờ.

## Ví dụ thực tế

Một ứng dụng đăng nhập dùng câu SQL ghép chuỗi trực tiếp từ input. Attacker nhập một chuỗi đặc biệt vào ô username khiến database trả về danh sách tài khoản. Nếu ứng dụng dùng parameterized queries, input sẽ được coi là dữ liệu chứ không phải câu lệnh SQL, làm giảm nguy cơ SQL injection.

## Cần ghi nhớ

- Web browsing không thể bị chặn hoàn toàn trong doanh nghiệp, nên phải kiểm soát và giám sát.
- HTTPS bảo vệ đường truyền, nhưng không đảm bảo nội dung website an toàn.
- Log HTTP là nguồn quan trọng để phát hiện scan, redirect và lỗi ứng dụng.
- Email security cần kết hợp kỹ thuật và đào tạo người dùng.
- SQL injection và XSS đều bắt nguồn từ việc xử lý input/output không an toàn.
