# 23.2 - Kỹ Thuật Kiểm Thử Bảo Mật Mạng (Network Security Testing Techniques)

## Tổng quan

Bài này trình bày các **kỹ thuật kiểm thử bảo mật mạng** từ góc độ người bảo vệ. Hiểu được kỹ thuật tấn công giúp defender biết mình đang phòng thủ gì và cách kiểm tra hiệu quả của các biện pháp bảo vệ.

---

## Ý chính

- Có nhiều loại network test khác nhau với mục đích và phạm vi khác nhau
- Operations security (OpSec) quan trọng kể cả khi đang test hệ thống của mình
- Kiểm tra định kỳ và sau mỗi thay đổi lớn là thực hành tốt
- Kết quả test phải được ghi chép, phân tích, và áp dụng cải thiện
- Mọi test phải có scope, authorization, và Rules of Engagement (ROE) rõ ràng

---

## Thuật ngữ quan trọng

| Thuật ngữ | Ý nghĩa |
|-----------|---------|
| Network security testing | Quá trình kiểm tra bảo mật của hạ tầng mạng |
| Operations security (OpSec) | Nguyên tắc bảo mật trong quá trình vận hành và test |
| Port scanning | Quét để phát hiện port nào đang mở trên một host |
| Service enumeration | Xác định phần mềm và phiên bản đang chạy trên các port |
| Banner grabbing | Thu thập thông tin phiên bản từ banner dịch vụ |
| Network mapping | Xây dựng bản đồ topology mạng |
| Passive reconnaissance | Thu thập thông tin mà không gửi traffic trực tiếp đến target |
| Active reconnaissance | Thu thập thông tin bằng cách gửi traffic đến target |
| ROE | Rules of Engagement - quy tắc thực hiện bài test |
| Scope | Phạm vi target được phép trong bài test |

## Ghi chú chi tiết

### Các loại Network Security Test

**1. Vulnerability Scanning**
- Tự động, dựa trên signature
- Tìm lỗ hổng đã biết
- Ít xâm phạm, ít rủi ro gián đoạn dịch vụ
- Cần lọc false positive

**2. Port và Service Scanning**
- Xác định attack surface
- Phát hiện service không cần thiết đang chạy
- Nền tảng cho các loại test khác

**3. Network Mapping**
- Xây dựng bản đồ topology mạng
- Phát hiện thiết bị không rõ nguồn gốc (rogue devices)
- Xác nhận segmentation đang hoạt động đúng

**4. Configuration Review**
- Kiểm tra cấu hình firewall, router, switch
- So sánh với baseline hoặc hardening guide
- Phát hiện misconfiguration

**5. Packet Analysis**
- Bắt và phân tích traffic mạng
- Phát hiện dữ liệu nhạy cảm truyền không mã hóa
- Xác nhận mã hóa đang hoạt động đúng

**6. Log Review**
- Phân tích log để tìm pattern bất thường
- Xác nhận logging đang đầy đủ
- Correlate event từ nhiều nguồn

### Operations Security (OpSec) khi kiểm thử

OpSec là tập hợp nguyên tắc để đảm bảo hoạt động test không gây ra rủi ro không cần thiết:

**Nguyên tắc OpSec trong testing:**

1. **Minimize footprint**: Chỉ làm những gì cần thiết, không mở rộng hơn scope
2. **Document everything**: Ghi chép mọi hoạt động, thời gian, và kết quả
3. **Communicate**: Thông báo cho các bên liên quan trước khi test
4. **Prepare rollback**: Có kế hoạch hoàn nguyên nếu test gây sự cố
5. **Protect test credentials**: Tài khoản dùng để test cần được bảo vệ và xóa sau khi xong

**Tại sao OpSec quan trọng kể cả khi test nội bộ?**
- Scan mạnh có thể làm gián đoạn dịch vụ production
- Tài khoản test với quyền cao là mục tiêu hấp dẫn
- Ghi chép giúp chứng minh tính hợp pháp nếu bị điều tra

### Phân loại test theo cấu hình

**Theo phạm vi:**
- Internal: từ bên trong mạng (sau firewall)
- External: từ internet (qua firewall, DMZ)
- Hybrid: cả hai

**Theo cấu hình kiến thức:**
- Black box: tester không biết trước về hệ thống (giống attacker thật)
- White box: tester biết toàn bộ về hệ thống (tối ưu hóa coverage)
- Gray box: tester biết một số thông tin (thực tế nhất)

**Theo tần suất:**
- One-time: cho một project hoặc certification cụ thể
- Periodic: định kỳ (quarterly, annually)
- Continuous: giám sát liên tục (monitoring, SIEM)

### Áp dụng kết quả test

Kết quả test có giá trị nhất khi được áp dụng đúng cách:

**Framework prioritization:**
```
Severity x Exploitability x Business Impact = Priority Score
```

**Ví dụ:**
- Lỗ hổng Critical nhưng không exploitable từ internet -> Priority thấp hơn
- Lỗ hổng Medium nhưng trên payment system -> Priority cao hơn

**Reporting checklist:**
- Mô tả lỗ hổng và nguyên nhân
- Cách exploit (ở mức đủ để remediate, không phải hướng dẫn tấn công)
- Impact nếu bị khai thác
- Khuyến nghị khắc phục cụ thể
- Timeline đề xuất

## Lệnh / Ví dụ minh họa

> **Quan trọng**: Chỉ thực hiện trên thiết bị và mạng thuộc quyền sở hữu hoặc có sự cho phép rõ ràng bằng văn bản.

### Kiểm tra kết nối và topology (lab)

```bash
# Kiểm tra kết nối đến gateway lab
ping -c 4 192.168.1.1        # Linux/Mac
ping -n 4 192.168.1.1        # Windows

# Traceroute xem path đến host trong lab
traceroute 192.168.1.100     # Linux/Mac
tracert 192.168.1.100        # Windows

# Xem ARP cache (các host đã giao tiếp gần đây)
arp -a
```

### Xem thông tin mạng của máy mình

```bash
# Xem IP, subnet, gateway (Windows)
ipconfig /all

# Xem IP, subnet, gateway (Linux)
ip addr show
ip route show

# Xem kết nối hiện tại
netstat -an      # Windows và Linux
ss -tunap        # Linux (nhanh hơn)
```

### Packet capture trên interface của lab (cần quyền admin/root)

```bash
# Bắt gói tin trên interface eth0 (Linux - lab interface)
# Chỉ dùng trên interface của máy mình, trong lab
tcpdump -i eth0 -n -c 100

# Bắt và lưu file để phân tích trong Wireshark
tcpdump -i eth0 -w /tmp/capture.pcap

# Filter chỉ xem HTTP traffic trong lab
tcpdump -i eth0 -n port 80
```

## Ví dụ thực tế

**Tình huống: Kiểm tra segmentation sau khi triển khai VLAN mới**

Scenario: Team mạng vừa tạo VLAN 30 cho guest Wi-Fi. Cần xác nhận guest không thể truy cập vào VLAN 10 (server).

*Với sự cho phép từ network manager:*

1. Kết nối vào guest Wi-Fi (VLAN 30)
2. Ping thử đến server trong VLAN 10 (`ping 10.0.10.1`)
3. Kết quả mong muốn: Request time out (firewall chặn)
4. Nếu ping thành công: Segmentation đang bị lỗi, cần kiểm tra ACL

*Ghi chép:* Thời gian test, người thực hiện, kết quả, và kết luận.

---

## Cần ghi nhớ

- Scope và authorization bằng văn bản là điều kiện tiên quyết
- Black/White/Gray box test phục vụ mục đích khác nhau
- OpSec bảo vệ cả quá trình test khỏi gây hại ngoài ý muốn
- Kết quả test phải có ưu tiên rõ ràng theo business impact
- Ghi chép đầy đủ là yêu cầu chuyên nghiệp, không phải tùy chọn
