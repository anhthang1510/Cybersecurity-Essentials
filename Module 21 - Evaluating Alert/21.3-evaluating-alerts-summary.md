# 21.3 - Tóm Tắt Module (Evaluating Alerts Summary)

## Tổng quan

Bài tóm tắt này hệ thống lại toàn bộ kiến thức của Module 21, giúp người học củng cố hiểu biết về nguồn gốc alert, công cụ phát hiện và phân tích, cũng như quy trình đánh giá alert có hệ thống.

---

## Ý chính

- Alert là tín hiệu đầu vào, không phải kết luận cuối cùng
- Nhiều nguồn và nhiều công cụ cùng tạo ra dữ liệu cho analyst
- Phân loại alert đúng (TP/FP/TN/FN) là kỹ năng cốt lõi
- Security Onion cung cấp nền tảng tích hợp cho NSM
- Snort rule là nền tảng của signature-based detection

---

## Thuật ngữ quan trọng

| Thuật ngữ | Ý nghĩa |
|-----------|---------|
| Alert | Cảnh báo từ hệ thống phát hiện |
| Security Onion | Nền tảng NSM tích hợp, nhiều công cụ |
| Snort | IDS/IPS rule-based mã nguồn mở |
| IDS | Hệ thống phát hiện xâm nhập (passive) |
| IPS | Hệ thống ngăn chặn xâm nhập (active) |
| SIEM | Tổng hợp và tương quan sự kiện bảo mật |
| False positive | Alert sai - hoạt động bình thường bị báo nguy |
| False negative | Bỏ sót - mối đe dọa thật không được phát hiện |
| Deterministic | Phân tích quy tắc cố định |
| Probabilistic | Phân tích xác suất và hành vi |
| NSM | Network Security Monitoring |
| Alert fatigue | Mệt mỏi do quá nhiều alert |

## Ghi chú chi tiết

### Tổng kết: Nguồn gốc alert (Module 21.1)

Alert đến từ nhiều tầng của hệ thống bảo mật:

```
[Network Traffic]
      |
      v
[IDS/IPS - Snort/Suricata] --> Alert (signature match)
[Zeek/Bro]                 --> Connection logs
[Firewall]                 --> Block/allow logs
      |
      v
[SIEM/Security Onion]      --> Aggregated alerts + correlation
      |
      v
[Analyst - Sguil/Squert]   --> Triage và đánh giá
```

**Công cụ phát hiện vs Công cụ phân tích:**

| Loại | Công cụ | Chức năng |
|------|---------|-----------|
| Phát hiện | Snort, Suricata | Sinh alert khi khớp rule |
| Phát hiện | OSSEC/Wazuh | Giám sát host, integrity check |
| Phân tích | Zeek/Bro | Log chi tiết mọi kết nối |
| Phân tích | Wireshark | Phân tích gói tin sâu |
| Phân tích | Sguil/Squert | Giao diện triage cho analyst |
| Tổng hợp | Security Onion | Tích hợp tất cả trên |

### Tổng kết: Snort Rule

Cấu trúc rule Snort:
```
[action] [protocol] [src_ip] [src_port] [direction] [dst_ip] [dst_port] ([options])
```

Ví dụ từ lab:
```
alert icmp any any -> 192.168.1.0/24 any (msg:"ICMP Ping Detected"; sid:1000002; rev:1;)
```

Các thành phần quan trọng:
- **action**: alert, log, pass, drop, reject
- **direction**: `->` (một chiều) hoặc `<>` (hai chiều)
- **msg**: thông điệp hiển thị trong alert
- **sid**: Snort ID, phải duy nhất (>1000000 cho custom rules)
- **content**: pattern cần khớp trong payload
- **rev**: số phiên bản rule

### Tổng kết: Đánh giá alert (Module 21.2)

**Ma trận phân loại:**

```
                        THỰC TẾ
                   Nguy hiểm | An toàn
PHÁT HIỆN  Có |    TP (tốt)  | FP (cần giảm)
           Không | FN (nguy!) | TN (tốt)
```

**Quy trình 5 bước:**
1. Xác nhận alert hợp lệ
2. Thu thập ngữ cảnh (IP, thời gian, lịch sử)
3. Phân tích dữ liệu bổ sung (Zeek log, PCAP, threat intel)
4. Phân loại (TP/FP/cần leo thang)
5. Hành động và ghi chép

**Hai phương pháp phân tích:**
- **Deterministic**: rule cứng, kết quả rõ ràng, phù hợp signature
- **Probabilistic**: ngữ cảnh + xác suất, phát hiện anomaly

## Ví dụ thực tế

**Kịch bản tổng hợp:**

Analyst thấy alert "Port Scan Detected" vào lúc 2:00 sáng.

*Bước đánh giá:*
1. Xác nhận: rule Snort `1000042` đang hoạt động bình thường
2. Ngữ cảnh: IP nguồn `203.x.x.x` (ngoài mạng), IP đích là web server công cộng, lúc 2h sáng không có lịch maintenance
3. Zeek log: 500 SYN packet đến 500 port khác nhau trong 30 giây, không có kết nối thành công
4. Phân loại: **True Positive** - đây là port scan thật
5. Hành động: Leo thang lên Tier 2, thêm IP vào blocklist tạm thời, ghi ticket

## Cần ghi nhớ

**7 điểm quan trọng nhất của Module 21:**

1. Alert là tín hiệu cần xác nhận, không phải kết luận
2. Security Onion tích hợp detection + collection + analysis
3. Snort rule: header (action + proto + addr + port) + options (msg, sid, content)
4. Custom Snort rule sid bắt đầu từ 1,000,000
5. False negative nguy hiểm hơn false positive
6. Quy trình đánh giá 5 bước giúp tránh sai sót
7. Ngữ cảnh (context) là yếu tố quyết định trong phân tích probabilistic
