# 21.2 - Tổng Quan Đánh Giá Alert (Overview of Alert Evaluation)

## Tổng quan

Không phải mọi alert đều cần phản ứng giống nhau. Kỹ năng quan trọng nhất của một analyst là khả năng **đánh giá và phân loại** alert nhanh chóng và chính xác. Bài này trình bày hai phương pháp phân tích chính và hệ thống phân loại alert.

---

## Ý chính

- Hai phương pháp phân tích: **deterministic** (dựa trên quy tắc cố định) và **probabilistic** (dựa trên xác suất và ngữ cảnh)
- Alert phải được phân loại thành: true positive, false positive, true negative, false negative
- Mục tiêu đánh giá là giảm false positive và không bỏ sót true positive
- Quy trình đánh giá alert cần có hệ thống, không dựa vào cảm tính
- Context (ngữ cảnh) đóng vai trò quyết định trong việc phân loại alert

---

## Thuật ngữ quan trọng

| Thuật ngữ | Ý nghĩa |
|-----------|---------|
| True positive (TP) | Alert đúng: mối đe dọa thật được phát hiện chính xác |
| False positive (FP) | Alert sai: hoạt động bình thường bị báo nguy hiểm |
| True negative (TN) | Không alert đúng: hoạt động bình thường, không cảnh báo |
| False negative (FN) | Bỏ sót: mối đe dọa thật nhưng không được phát hiện |
| Deterministic | Phân tích theo rule cố định: kết quả luôn giống nhau với cùng input |
| Probabilistic | Phân tích theo xác suất, hành vi, ngữ cảnh: linh hoạt hơn |
| Alert classification | Quá trình phân loại alert theo mức độ và loại |
| Benign trigger | Hoạt động hợp lệ vô tình kích hoạt alert (gây false positive) |

---

## Ghi chú chi tiết

### Ma trận phân loại alert

```
                    | Thực sự nguy hiểm | Thực sự an toàn
--------------------|-------------------|------------------
Alert được sinh ra  | TRUE POSITIVE (TP)| FALSE POSITIVE (FP)
Không có alert      | FALSE NEGATIVE (FN)| TRUE NEGATIVE (TN)
```

**Ý nghĩa thực tế:**
- **TP**: Mục tiêu lý tưởng. Analyst phát hiện đúng tấn công.
- **FP**: Gây alert fatigue. Cần tinh chỉnh rule.
- **FN**: Nguy hiểm nhất. Attacker hoạt động mà không bị phát hiện.
- **TN**: Trạng thái bình thường mong muốn.

### Phân tích Deterministic

Phân tích deterministic dựa trên **quy tắc cố định và rõ ràng**. Kết quả hoàn toàn có thể dự đoán:

**Đặc điểm:**
- Kết quả nhất quán: cùng input luôn cho cùng output
- Dễ giải thích và kiểm tra
- Phù hợp với signature-based detection
- Hạn chế: không phát hiện được tấn công chưa có signature

**Ví dụ deterministic:**
- Rule Snort: "Nếu thấy payload chứa `cmd.exe` trong HTTP request -> alert"
- Firewall rule: "Nếu IP nguồn trong blacklist -> drop và log"
- Nếu điều kiện đúng -> alert. Luôn luôn.

### Phân tích Probabilistic

Phân tích probabilistic xem xét **nhiều yếu tố và ngữ cảnh** để đánh giá xác suất đây là mối đe dọa:

**Đặc điểm:**
- Linh hoạt hơn, phát hiện được anomaly
- Cần nhiều dữ liệu ngữ cảnh hơn
- Kết quả là xác suất, không phải kết luận tuyệt đối
- Phù hợp với behavior-based và machine learning detection

**Ví dụ probabilistic:**
- Người dùng này thường đăng nhập từ Hà Nội lúc 8h sáng. Hôm nay đăng nhập từ Berlin lúc 3h sáng -> xác suất cao là bất thường
- Traffic đến server tăng đột biến 500% so với baseline -> có thể là DDoS

### Quy trình đánh giá alert

**Bước 1: Xác nhận alert**
- Alert có hợp lệ không? Rule có đang hoạt động đúng không?
- Đây có phải là loại alert hay gặp false positive không?

**Bước 2: Thu thập ngữ cảnh**
- IP nguồn/đích là gì? Thuộc mạng nào?
- Thời điểm xảy ra có bất thường không?
- Có alert tương tự gần đây không?

**Bước 3: Phân tích dữ liệu bổ sung**
- Xem Zeek log: kết nối thành công chưa? Bao nhiêu bytes trao đổi?
- Xem packet capture nếu cần: nội dung thực sự là gì?
- Kiểm tra threat intelligence: IP đó có trong danh sách xấu không?

**Bước 4: Phân loại**
- Kết luận: TP, FP, hay cần leo thang (escalate)?

**Bước 5: Hành động và ghi chép**
- Nếu TP: theo quy trình incident response
- Nếu FP: ghi nhận để cải thiện rule sau
- Luôn ghi chép quá trình đánh giá

### Tại sao false negative nguy hiểm hơn false positive?

False positive gây phiền toái và tốn thời gian. False negative để **attacker hoạt động tự do mà không bị phát hiện**. Đó là lý do các hệ thống bảo mật thường được tinh chỉnh để giảm false negative trước, dù điều đó có thể tăng false positive tạm thời.

---

## Ví dụ thực tế

**Tình huống 1 - False positive:**
Nhóm IT chạy vulnerability scan định kỳ trên mạng nội bộ vào tối thứ Sáu. IDS phát hiện port scan và tạo alert "Possible Reconnaissance". Analyst kiểm tra: IP nguồn là máy quản lý của IT, scan được lịch sẵn. Kết luận: **false positive** - hoạt động hợp lệ.

**Tình huống 2 - True positive:**
Alert "SQL Injection attempt" trên web server. Analyst xem log: payload chứa `' OR '1'='1`, IP nguồn không rõ nguồn gốc, không có lịch scan. Zeek log cho thấy nhiều request tương tự trong 5 phút. Kết luận: **true positive** - cần leo thang.

---

## Cần ghi nhớ

- Bốn kết quả phân loại: TP, FP, TN, FN
- FN nguy hiểm hơn FP trong bối cảnh bảo mật
- Deterministic = quy tắc cố định; Probabilistic = xác suất + ngữ cảnh
- Đánh giá alert luôn cần ngữ cảnh, không chỉ nhìn vào alert đơn lẻ
- Ghi chép quá trình đánh giá giúp cải thiện hệ thống lâu dài
