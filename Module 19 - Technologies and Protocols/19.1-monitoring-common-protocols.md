# 19.1 - Monitoring Common Protocols

## Tổng quan

Phần này nói về việc quan sát các protocol phổ biến để hiểu hành vi mạng. Mỗi protocol để lại dấu vết khác nhau; khi biết dữ liệu nào cần xem, analyst có thể phát hiện bất thường nhanh hơn.

## Ý chính

- DNS logs giúp phát hiện domain lạ, malware callback hoặc DNS tunneling.
- HTTP/HTTPS logs cho biết user hoặc host truy cập tài nguyên web nào.
- TLS metadata và certificate information vẫn hữu ích dù payload được mã hóa.
- DHCP logs giúp ánh xạ IP với MAC/device tại thời điểm cụ thể.
- Authentication logs giúp phát hiện brute force, impossible travel hoặc privilege abuse.
- Email protocol logs hỗ trợ điều tra phishing, spoofing và suspicious attachments.
- Remote access logs như SSH, RDP, VPN cho biết ai truy cập từ đâu, khi nào.

## Thuật ngữ quan trọng

| Thuật ngữ | Ý nghĩa |
|---|---|
| DNS query | Yêu cầu phân giải tên miền thành IP hoặc bản ghi khác. |
| HTTP request | Yêu cầu web từ client tới server. |
| TLS metadata | Thông tin handshake/certificate có thể quan sát mà không cần decrypt payload. |
| DHCP lease | Bản cấp phát IP cho thiết bị trong một khoảng thời gian. |
| Authentication log | Log ghi nhận đăng nhập, thất bại, thành công hoặc thay đổi quyền. |
| Beaconing | Hành vi kết nối định kỳ tới máy chủ điều khiển hoặc dịch vụ ngoài. |
| DNS tunneling | Kỹ thuật giấu dữ liệu hoặc command trong DNS queries/responses. |
| Baseline | Mẫu hành vi bình thường dùng để so sánh bất thường. |

## Ghi chú chi tiết

**DNS monitoring:**

DNS là nguồn dữ liệu rất hữu ích vì nhiều malware cần phân giải domain để liên lạc command-and-control. Query tới domain mới tạo, chuỗi domain dài bất thường hoặc tần suất query đều có thể là tín hiệu cần điều tra.

**HTTP/HTTPS monitoring:**

Với HTTP, analyst có thể xem host, URI, user-agent, status code và phương thức. Với HTTPS, payload được mã hóa nên cần dựa nhiều hơn vào metadata, proxy logs, SNI, certificate, flow và endpoint telemetry.

**DHCP và asset context:**

IP address có thể thay đổi theo thời gian. DHCP logs giúp biết tại thời điểm sự cố IP đó thuộc về thiết bị nào. Nếu thiếu DHCP hoặc asset inventory, điều tra sẽ khó quy trách nhiệm chính xác.

**Authentication monitoring:**

Nhiều cuộc tấn công bắt đầu từ credential abuse. Log đăng nhập thất bại nhiều lần, đăng nhập từ vị trí lạ, thay đổi MFA hoặc tạo tài khoản mới đều cần được chú ý.

**Protocol context:**

Không phải mọi traffic lạ đều độc hại. Analyst cần hiểu baseline: server DNS nói chuyện với DNS resolver là bình thường, nhưng máy user gửi hàng nghìn query TXT dài có thể đáng ngờ.

## Ví dụ thực tế

**Ví dụ 1: Phát hiện malware bằng DNS**

Một endpoint liên tục query domain có chuỗi ngẫu nhiên mỗi 60 giây. DNS logs kết hợp với endpoint logs cho thấy tiến trình lạ tạo kết nối định kỳ. Đây có thể là beaconing tới command-and-control.

**Ví dụ 2: Điều tra IP bằng DHCP logs**

Firewall log cho thấy `192.168.10.45` truy cập server bị cấm lúc 10:15. DHCP logs cho biết IP này lúc đó thuộc laptop của user A, giúp analyst tiếp tục điều tra đúng thiết bị.

**Ví dụ 3: Brute force VPN**

VPN logs ghi nhận hàng trăm lần đăng nhập thất bại từ một IP nước ngoài. SIEM correlation với account lockout logs giúp xác định đây là brute force attempt.

## Cần ghi nhớ

- **DNS là nguồn phát hiện mạnh**: nhiều malware để lại dấu vết qua domain queries.
- **HTTPS không làm monitoring vô dụng**: metadata và logs vẫn có giá trị.
- **DHCP giúp gắn IP với thiết bị**: rất quan trọng khi IP động.
- **Authentication logs phản ánh identity risk**: brute force và abuse thường xuất hiện ở đây.
- **Protocol logs cần ngữ cảnh**: phải so sánh với baseline và asset role.
